abstract: |
  Most programming languages allow any function to access ambient state---globals, system calls, built-in functions---undermining local reasoning, testability, and security. \emph{Dependency injection} mitigates these problems but only as a design pattern. We propose \textbf{hermeticity} as a first-class semantic property of functions: a function is hermetic iff it accesses existing state only through its parameters. A hermetic main function enforces inversion of control at the language level and eliminates \emph{ambient authority}, satisfying the defining requirement of capability-based security. Hermeticity is orthogonal to purity: the former restricts \emph{access} to state, the latter restricts \emph{interaction}. We formalize both notions in terms of \emph{live} and \emph{inert} values, where a value is live iff it provides retargetable access to existing state---a property we make precise via a \textbf{mockability test}. Building a hermetic programming language requires an inert ambient scope: standard libraries export interfaces and inert constructors, while concrete resources are injected by the runtime. Behavioral referential transparency, composability, testability, determinism, portability, and capability-based security all follow from a single semantic rule: \emph{a function may only access existing state through its parameters}. We discuss how hermeticity applies in purely functional languages such as Haskell, and survey existing systems (Joe-E, \texttt{cap-std}, Wuffs, SES, WASI) already moving in this direction.
